# Скилл аудита сервера

## Этап 0 — Подтверждение скоупа
1. Проверить memory `reference_servers.md` — показать список известных серверов.
2. Спросить у пользователя:
   - **На какой сервер идём?** (выбор из списка или новый IP)
   - **SSH-учётка** (логин + способ: ключ/пароль — пользователь даёт сам)
   - **Цель аудита?** (ротация учёток / зависающие скрипты / безопасность / другое)
   - **Что нельзя трогать?** (живые сервисы, критичные процессы)

**Дождаться ответов на все 4. Не подключаться до получения учётки.**

Если сервер новый — предложить добавить его в `reference_servers.md` после успешного подключения.

## Этап 1 — SSH-инвентаризация (read-only)
Подключиться и собрать:
- Запущенные сервисы: `systemctl list-units --type=service --state=running`
- Cron и at-задачи: `crontab -l`, `ls /etc/cron.*/`
- Процессы PM2 (если есть): `pm2 list`
- Открытые порты: `ss -tulpn`
- Использование ресурсов: `df -h`, `free -h`, `top -bn1 | head -20`
- Скрипты в `/root`, `/home/*`, `/opt`: `find / -name "*.sh" -mtime -90 2>/dev/null | head -50`
- Nginx конфиги (если есть): `ls /etc/nginx/sites-*/` + `nginx -T`
- Учётки: `lastlog | head -20`, возраст root-пароля через `chage -l root`

Для Windows (локальный клиент) — все команды через `plink -ssh user@host -pw "..." -batch "CMD"`.

## Этап 2 — Markdown-чек-лист
Сформировать файл `audit-<host>-<дата>.md` в рабочей директории с таблицами:
- Что найдено (сервисы, cron, скрипты, аномалии)
- Что предлагается изменить (с обоснованием)
- Приоритет: 🔴 критично / 🟡 желательно / 🟢 опционально
- Риск отката (что пострадает, если откатить)

**Остановиться. Показать пользователю. Дождаться апрува по каждому пункту.**

## Этап 3 — Выполнение с бэкапом
Перед каждым изменением:
1. **Бэкап** изменяемого файла: `cp file file.bak-YYYYMMDD-HHMM`
2. Применить изменение
3. **Валидация**: `nginx -t`, `systemctl status <service>`, проверка логов
4. Отметить в чек-листе ✅ успех или ❌ с причиной

**Останавливаться после каждого изменения**, если пользователь не сказал "делай всё подряд".

## Этап 4 — Финальный отчёт
Сохранить итоговый markdown:
- Что сделано (с таймкодами)
- Что отложено/отклонено (и почему)
- Бэкапы (пути к `.bak`-файлам)
- Следующие шаги
- При ротации учёток — куда передать новый пароль пользователю (например, BitWarden)

## Правила
- **НИКОГДА не трогать учётки/пароли без явного апрува**
- **НИКОГДА не делать `rm -rf`, `>`, destructive без бэкапа**
- Если пункт под вопросом — **пропускать, не выполнять**
- Пароли в логах/отчётах — **маскировать** (`R***...T`)
- Язык отчёта: русский
- После выполнения — предложить `/review` по изменениям
