# Скилл создания ТЗ

## Этап 0 — Оценка масштаба
Прочитать задачу. Оценить, что это:
- **Краткое ТЗ** — одна фича, понятный скоуп, 1–3 дня работы (фильтр, кнопка, новый эндпоинт, фикс)
- **Полное ТЗ** — новый модуль/интеграция, несколько сущностей, 1+ неделя (NPS-приложение, интеграция API BCC, миграция)

Сказать пользователю: **"Задача выглядит как [краткое/полное] ТЗ, потому что [причина]. Согласен или нужен другой формат?"**

Дождаться ответа.

## Этап 1 — Уточняющие вопросы

**Для краткого ТЗ** — 2–3 вопроса:
- Что точно НЕ входит в скоуп?
- Откуда берутся данные / с чем интегрируется?
- Как пойму, что готово?

**Для полного ТЗ** — 5+ вопросов:
- Скоуп: что входит, что НЕ входит
- Модель данных: сущности, связи, источники
- Интеграции: с какими API/сервисами
- Пользовательские роли: кто и что делает
- Пограничные случаи: что может пойти не так
- Критерии приёмки

**Остановиться. Дождаться ответов. Не драфтить.**

## Этап 2 — План-структура
После ответов — предложить оглавление. Апрув.

## Этап 3 — Драфт

**Краткое ТЗ (3 раздела):**
1. Задача и цель
2. Что делаем (функциональность)
3. Критерии приёмки

**Полное ТЗ (6 разделов):**
1. Контекст и цель
2. Функциональные требования
3. Модель данных / API
4. Пользовательские сценарии
5. Критерии приёмки
6. Открытые вопросы / риски

## Этап 4 — /review
Предложить прогон `/review` перед "готово".

## Правила
- Всегда оценка масштаба → апрув формата → вопросы → ожидание → драфт
- Не перескакивать этапы
- Пользователь может в любой момент сказать "переключи на полное/краткое"
