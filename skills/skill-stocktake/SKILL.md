## Quick scan

1. Прочитай `results.json` (Read).
2. Через Glob найди все `SKILL.md` в обеих папках, сравни с записанными в `results.json`: новые файлы, исчезнувшие, и те, чей размер/содержимое отличается от закэшированного (сравнивай по полю `content_hash` — это первые ~40 символов тела после frontmatter, см. схему ниже; если для скилла хэша в кэше нет — считай изменённым).

4. Переоцени только изменённые по тем же критериям фазы 2.
5. Неизменённые перенеси из старого `results.json` без переоценки.
6. Покажи только дифф (что переоценено).
7. Сохрани обновлённый `results.json` (Write).

### Фаза 2 — Оценка качества

Раздели скиллы на батчи по ~15–20 штук. Для каждого батча запусти субагента (Agent, subagent_type `general-purpose`), передав ему список путей скиллов, чек-лист и критерии вердиктов ниже. Субагент читает каждый скилл и возвращает по нему JSON:

`{ "verdict": "Keep"|"Improve"|"Update"|"Retire"|"Merge into [X]", "reason": "..." }`

После каждого батча сохраняй промежуточный результат в `results.json` со `status: "in_progress"`. Если при старте увидел `status: "in_progress"` — продолжай с первого неоценённого скилла.

**Чек-лист для каждого скилла:**

```
- [ ] Пересечение по содержанию с другими скиллами проверено
- [ ] Пересечение с MEMORY.md / CLAUDE.md проверено
- [ ] Актуальность тех. ссылок проверена (WebSearch, если есть имена тулов / флаги CLI / версии API)
- [ ] Частота использования учтена (по своему опыту сессий, если есть)
```

**Критерии вердиктов:**

| Вердикт | Значение |
|---------|----------|
| Keep | Полезен и актуален |
| Improve | Стоит оставить, но нужны конкретные правки |
| Update | Технология/ссылки устарели (проверь WebSearch) |
| Retire | Низкое качество, протух, или польза не оправдывает вес |
| Merge into [X] | Существенное пересечение с другим скиллом; назови цель слияния |

Оценка — **целостное суждение**, не числовой балл. Ориентиры: *применимость* (есть ли конкретные шаги/команды/примеры), *соответствие скоупа* (имя, триггер и содержание согласованы), *уникальность* (нельзя заменить MEMORY.md / CLAUDE.md / другим скиллом), *актуальность* (ссылки работают в текущем окружении).

**Требования к полю `reason`** — оно должно быть самодостаточным и позволять принять решение:
- Не писать просто «без изменений» — всегда повторяй ключевое обоснование.
- **Retire**: укажи (1) какой конкретно дефект, (2) что закрывает ту же потребность вместо него.
- **Merge**: назови цель и что именно перенести.
- **Improve**: какую секцию/что изменить, при необходимости — целевой размер.
- **Keep** (в quick scan при изменении только mtime): повтори исходное обоснование, не пиши «не изменилось».

### Фаза 4 — Консолидация (только по апруву)

1. **Retire / Merge**: по каждому файлу покажи подробное обоснование до подтверждения — какой дефект, чем закрывается потребность, что зависит от этого скилла (другие скиллы, ссылки в MEMORY.md / CLAUDE.md, триггеры в CLAUDE.md). **При удалении/слиянии скилла синхронно предложи почистить его строку в таблице скиллов и триггеры в обоих CLAUDE.md.**
2. **Improve**: покажи конкретные предложения с обоснованием. Решает пользователь.
3. **Update**: покажи обновлённое содержание с проверенными источниками.
4. Проверь число строк MEMORY.md; если >100 — предложи сжать.

**Удаление и слияние всегда требуют явного подтверждения пользователя.**

## Схема results.json

`evaluated_at` — фактическое UTC-время завершения. Получить через PowerShell: `(Get-Date).ToUniversalTime().ToString("yyyy-MM-ddTHH:mm:ssZ")`. Не использовать дату без времени.

```json
{
  "evaluated_at": "2026-06-11T10:00:00Z",
  "mode": "full",
  "batch_progress": { "total": 37, "evaluated": 37, "status": "completed" },
  "skills": {
    "skill-name": {
      "path": "C:\\Users\\umidi\\.claude\\skills\\skill-name\\SKILL.md",
      "verdict": "Keep",
      "reason": "Конкретное, применимое, уникальная польза для X",
      "content_hash": "первые ~40 символов тела после frontmatter"
    }
  }
}
```
