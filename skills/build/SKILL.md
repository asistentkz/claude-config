## Принципы оркестрации

**Каждый этап = отдельный subagent через Agent tool.**

Зачем: свежий контекст на каждом этапе предотвращает "context rot".
Оркестратор НЕ выполняет этапы сам — он формирует prompt, запускает Agent, получает structured summary, показывает пользователю, ждёт подтверждения.

**Цикл каждого этапа:**
1. Оркестратор формирует prompt для subagent (задача + summary предыдущих этапов)
2. Оркестратор запускает Agent с этим prompt
3. Subagent выполняет работу, возвращает structured summary
4. Оркестратор извлекает summary, показывает пользователю
5. Пользователь подтверждает / просит правки
6. Если правки — оркестратор запускает новый Agent с уточнениями
7. Summary сохраняется, переход к следующему этапу

**Оркестратор хранит в своём контексте ТОЛЬКО:**
- Исходную задачу ($ARGUMENTS)
- Тип задачи и список этапов
- Accumulated summaries (structured)
- Статус каждого этапа

**Оркестратор НЕ хранит:**
- Полный вывод subagent-ов (только summary)
- Содержимое файлов
- Детали реализации

## Формат structured summary

Каждый subagent ОБЯЗАН вернуть результат в этом формате:

```
STAGE_SUMMARY:
  stage: [номер и название]
  status: DONE | DONE_WITH_CONCERNS | BLOCKED
  decisions:
    - [ключевое решение 1]
    - [ключевое решение 2]
  artifacts:
    - [файл создан/изменён: путь — назначение]
  key_facts:
    - [факт который нужен следующим этапам]
  concerns:
    - [проблема или риск, если есть]
  next_stage_needs:
    - [что следующий этап должен знать]
```

**Правила передачи контекста subagent-у:**

Subagent получает prompt из 4 блоков:
1. **TASK** — исходное описание задачи (неизменно для всех этапов)
2. **SPEC** — summaries этапов 1-3 (БА, Архитектор, Docs) — передаются ПОЛНОСТЬЮ всем последующим этапам, это "спецификация проекта"
3. **PROGRESS** — summaries этапов 4+ — передаются сжато (только decisions + artifacts + concerns)
4. **STAGE_INSTRUCTIONS** — конкретные инструкции для текущего этапа

**Правило сжатия PROGRESS:**
- Из каждого summary этапа 4+ берём только: decisions, artifacts, concerns
- key_facts и next_stage_needs включаются в STAGE_INSTRUCTIONS если релевантны
- Это предотвращает линейный рост контекста

**Исключение:** Этап 12 (Финальный отчёт) получает ВСЕ summaries полностью.

## Шаг 0: Определение типа задачи

Проанализируй задачу и определи тип:

**Новый модуль** — нужен полный цикл от требований до деплоя
**Доработка** — изменение существующего функционала
**Багфикс** — поиск и исправление ошибки

Покажи пользователю:
```
Тип задачи: [тип]
Стек проекта: [определи из CLAUDE.md / package.json / go.mod / composer.json / pubspec.yaml]
Этапы которые нужны: [список]
Пропускаем: [список ненужных этапов и почему]
```
**Жди подтверждения перед стартом**

После подтверждения инициализируй:
```
BUILD_CONTEXT:
  task: $ARGUMENTS
  type: [определённый тип]
  stack: [стек проекта]
  stages: [список этапов]
  summaries: {}
```

## Этапы по типу задачи

### Новый модуль — все этапы
1. БА → 2. Архитектор → 3. Docs → 4. QA (тесты TDD) → 5. Backend → 6. Frontend → 7. QA (прогон) → 8. Security → 9. Review → 10. DevOps → 11. Docs (обновление) → 12. Финальный отчёт

### Доработка — без полного БА
1. Research → 2. Архитектор → 3. QA (тесты TDD) → 4. Backend → 5. Frontend (если нужно) → 6. QA (прогон) → 7. Review → 8. DevOps → 9. Docs (обновление)

### Багфикс — быстрый цикл
1. Диагност → 2. QA (тест на баг) → 3. Backend/Frontend → 4. QA (прогон) → 5. Review → 6. Docs (если затронута логика)

## Этап 1: Бизнес-аналитик

**Роль subagent**: Понять ЧТО нужно сделать и ЗАЧЕМ

**Оркестратор формирует prompt:**

```
Ты — Бизнес-аналитик. Твоя задача — сформулировать требования.

TASK: {$ARGUMENTS}
STACK: {стек проекта}

Выполни:
1. Сформулируй требования к задаче
2. Определи пользовательские сценарии (happy path + edge cases)
3. Найди зависимости от существующих модулей (используй Grep, Glob, Read)
4. Поищи аналогичные решения (WebSearch) если задача нетривиальная

ОБЯЗАТЕЛЬНО верни результат в формате:
STAGE_SUMMARY:
  stage: "1. Бизнес-аналитик"
  status: DONE | DONE_WITH_CONCERNS | BLOCKED
  decisions:
    - [каждое требование как решение]
  artifacts: []
  key_facts:
    - Требования: [список]
    - Сценарии: основной + edge cases
    - Зависимости: модули, таблицы, API
  concerns:
    - [блокеры или неясности, если есть]
  next_stage_needs:
    - [что Архитектор должен учесть]
```

**После получения результата:**
1. Извлеки STAGE_SUMMARY из ответа subagent
2. Сохрани в summaries["stage_1"]
3. Покажи пользователю summary в читаемом виде
4. **"Требования готовы. Переходим к архитектуре?"**

```
Ты — Технический писатель. Создай документацию ДО написания кода.
```

## Этап 4: QA — Тесты ДО кода (TDD)

**Роль subagent**: Написать тесты ПЕРЕД кодом

**Оркестратор формирует prompt:**

```
Ты — QA-инженер. Напиши тесты ДО написания кода (TDD подход).

TASK: {$ARGUMENTS}

SPEC:
{summaries["stage_1"] — полностью}
{summaries["stage_2"] — полностью}
{summaries["stage_3"] — полностью}

Подход TDD:
1. На основе требований и архитектуры — напиши тесты
2. Тесты описывают ЧТО должно работать, ещё без реализации
3. Все тесты будут падать (красные) — это нормально
4. Backend и Frontend потом напишут код чтобы тесты стали зелёными

Определи тестовый фреймворк по стеку проекта (Jest, PHPUnit, Go test, Flutter test и т.д.)

Напиши тесты на:
- Основные сценарии из требований БА
- Edge cases (пустые данные, ошибки, граничные значения)
- Интеграции (БД, API)

ОБЯЗАТЕЛЬНО верни результат в формате:
STAGE_SUMMARY:
  stage: "4. QA TDD"
  status: DONE | DONE_WITH_CONCERNS | BLOCKED
  decisions:
    - Фреймворк: [выбранный]
    - Подход: [что тестируем]
  artifacts:
    - [каждый файл тестов: путь — что покрывает]
  key_facts:
    - Тестов написано: X (все красные)
    - Покрытие сценариев: [список]
  concerns:
    - [что сложно протестировать]
  next_stage_needs:
    - [какие тесты должен пройти Backend]
```

**После получения результата:**
1. Извлеки STAGE_SUMMARY, сохрани в summaries["stage_4"]
2. Покажи количество тестов и покрытие
3. **"Тесты написаны (красные). Переходим к backend-коду?"**

## Этап 10: DevOps

```
{собрать concerns из всех summaries}
```

## Этап 12: Финальный отчёт

**Роль subagent**: Собрать итоговый отчёт по всем этапам

**ИСКЛЮЧЕНИЕ: этот subagent получает ВСЕ summaries полностью (без сжатия).**

**Оркестратор формирует prompt:**

```
Ты — Менеджер проекта. Собери финальный отчёт.

TASK: {$ARGUMENTS}

ВСЕ SUMMARIES (полностью):
{все summaries от stage_1 до stage_11 — без сжатия}

Собери итог по всем этапам:

Задача: [описание]
Тип: [новый модуль / доработка / багфикс]
Файлы создано: X
Файлы изменено: X
Тесты: X/Y зелёных
Безопасность: ок / проблемы
Code Review: вердикт
Готовность к деплою: ок / с оговорками / нет
Нерешённые concerns: [список или "нет"]

ОБЯЗАТЕЛЬНО верни результат в формате:

  status: DONE
  decisions:

  artifacts:

  key_facts:
    - [ключевые метрики]
  concerns:
    - [нерешённые проблемы]
  next_stage_needs: []
```

**После получения результата:**
1. Покажи пользователю итоговый отчёт
2. Работа завершена

## Правила оркестратора

1. **На каждом этапе спрашивай** — "Переходим к следующему?" / "Нужны правки?"
2. **Если пользователь говорит "пропусти"** — пропускай этап, сохрани в summaries как status: SKIPPED
3. **Если subagent вернул BLOCKED** — останови и сообщи сразу, не пытайся обойти
4. **Не передавай файлы в subagent** — передавай только summary и инструкции, subagent сам читает файлы
5. **Контекст передаётся ТОЛЬКО через summary** — НИКОГДА не копируй полный вывод subagent в prompt следующего
6. **Оркестратор НЕ кодит сам** — он формирует prompts, запускает Agent, парсит summary
7. **Если subagent вернул DONE_WITH_CONCERNS** — покажи concerns пользователю перед переходом
8. **Максимум 2 retry** — если subagent BLOCKED дважды на одном этапе, эскалируй к пользователю
9. **Этап 12 получает ВСЕ summaries полностью** — единственный этап без сжатия
10. **Не редактируй файлы** без показа чек-листа и одобрения пользователя

---

## Обработка статусов subagent

**DONE:** Извлеки summary, покажи пользователю, жди подтверждения.

**DONE_WITH_CONCERNS:** Покажи concerns отдельно. Спроси: "Решаем сейчас или двигаемся дальше?" Если решаем — запусти fix-subagent с конкретными concerns.

**BLOCKED:** Покажи причину блокировки. Варианты:
1. Не хватает контекста → дополни prompt, перезапусти
2. Задача слишком сложная → разбей этап на подэтапы
3. Конфликт с предыдущим этапом → вернись к предыдущему
4. Невозможно выполнить → эскалируй к пользователю

**SKIPPED:** Пользователь попросил пропустить. Сохрани summary с status: SKIPPED, decisions: ["Пропущен по решению пользователя"].

**Нет STAGE_SUMMARY в ответе:** Subagent забыл формат. Извлеки ключевую информацию из ответа, сформируй summary вручную, пометь concerns: ["summary извлечён оркестратором — возможна потеря деталей"].
