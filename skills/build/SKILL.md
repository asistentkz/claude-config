## Шаг 0: Определение типа задачи

Проанализируй задачу и определи тип:

**Новый модуль** — нужен полный цикл от требований до деплоя
**Доработка** — изменение существующего функционала
**Багфикс** — поиск и исправление ошибки

Покажи пользователю:
```
Тип задачи: [тип]
Стек проекта: [определи из CLAUDE.md / package.json / go.mod / composer.json / pubspec.yaml]
Этапы которые нужны: [список]
Пропускаем: [список ненужных этапов и почему]
```
**Жди подтверждения перед стартом**

## Этапы по типу задачи

### Новый модуль — все этапы
1. БА → 2. Архитектор → 3. Docs → 4. QA (тесты TDD) → 5. Backend → 6. Frontend → 7. QA (прогон) → 8. Security → 9. Review → 10. DevOps → 11. Docs (обновление)

### Доработка — без полного БА
1. Research → 2. Архитектор → 3. QA (тесты TDD) → 4. Backend → 5. Frontend (если нужно) → 6. QA (прогон) → 7. Review → 8. DevOps → 9. Docs (обновление)

### Багфикс — быстрый цикл
1. Диагност → 2. QA (тест на баг) → 3. Backend/Frontend → 4. QA (прогон) → 5. Review → 6. Docs (если затронута логика)

## Этап 1: Бизнес-аналитик

**Роль**: Понять ЧТО нужно сделать и ЗАЧЕМ

- Сформулировать требования к задаче
- Определить пользовательские сценарии (happy path + edge cases)
- Найти зависимости от существующих модулей (`/research`)
- Поискать аналогичные решения в интернете (WebSearch)

**Результат:**
```
Требования:
  - [ ] ...
Сценарии:
  - Основной: ...
  - Ошибка: ...
Зависимости: [модули, таблицы, API]
```
**"Требования готовы. Переходим к архитектуре?"**

---

## Этап 2: Архитектор

**Роль**: Спроектировать КАК это сделать

- Какие файлы создать / изменить (чек-лист!)
- Структура БД: новые таблицы, изменения существующих
- API эндпоинты (если нужно)
- Учесть архитектуру и стек проекта
- Проверить что не дублируем существующий код

**Результат:**
```
Файлы для создания:
  - [ ] путь/файл — назначение
Файлы для изменения:
  - [ ] путь/файл — что меняем
БД:
  - [ ] изменения схемы
Архитектурные решения:
  - ...
```
**"Архитектура готова. Документируем перед кодом?"**

---

## Этап 3: Документация (ДО кода)

**Роль**: Задокументировать что будем строить

На основе требований БА и решений Архитектора создать:
- **README модуля** — назначение, как работает, зависимости
- **Схема БД** — таблицы, поля, связи (если новые/изменённые)
- **API эндпоинты** — если есть
- **Бизнес-логика** — основные процессы, потоки данных

Сохранить в `_docs/[модуль]/`

**Зачем до кода:** документация описывает НАМЕРЕНИЕ. После кода — описывает реализацию. Если они расходятся — это сигнал что что-то пошло не так.

**"Документация создана. Переходим к тестам (TDD)?"**

---

## Этап 4: QA — Тесты ДО кода (TDD)

**Роль**: Написать тесты ПЕРЕД кодом — описать ожидаемое поведение

**Подход TDD:**
1. На основе требований БА и архитектуры — написать тесты
2. Тесты описывают ЧТО должно работать, ещё без реализации
3. Все тесты будут падать (красные) — это нормально
4. Backend и Frontend потом пишут код чтобы тесты стали зелёными

- Тестовый фреймворк — определить по стеку проекта (Jest, PHPUnit, Go test, Flutter test и т.д.)
- Тесты на основные сценарии из требований БА
- Тесты на edge cases (пустые данные, ошибки, граничные значения)
- Тесты на интеграции (БД, API)

**Результат:**
```
Тестов написано: X (все красные — реализации ещё нет)
Фреймворк: [определённый по стеку]
Покрытие сценариев:
  - [сценарий] — тест готов
  - [edge case] — тест готов
```
**"Тесты написаны. Переходим к backend-коду?"**

---

## Этап 5: Backend-разработчик

**Роль**: Написать серверный код чтобы тесты стали зелёными

- Следовать архитектуре из Этапа 2
- Цель: пройти тесты из Этапа 4
- Использовать безопасные практики (prepared statements, экранирование вывода)
- Не хардкодить credentials
- Логировать ошибки

**Показать код пользователю перед записью в файл!**

**После написания — прогнать тесты**

**"Backend готов, тесты зелёные. Переходим к frontend?"**

---

## Этап 6: Frontend-разработчик

**Роль**: Сверстать интерфейс и клиентскую логику

- Шаблоны / компоненты (по стеку проекта)
- Стили (учесть существующий стиль проекта)
- Валидация форм на клиенте + на сервере
- Экранирование вывода (XSS)
- UX: сообщения об ошибках, loading states

**Показать шаблоны/вёрстку пользователю перед записью!**

**"Frontend готов. Прогоняем все тесты?"**

---

## Этап 7: QA — Прогон тестов

**Роль**: Финальная проверка — все тесты должны быть зелёными

- Прогнать все тесты из Этапа 4
- Проверить интеграцию Backend + Frontend
- Проверить что существующий функционал не сломан

**Результат:**
```
Тесты: X/X зелёных
Регрессия: не затронуто / проблемы
Ручная проверка: [чек-лист]
```
**"Все тесты зелёные. Проверяем безопасность?"**

---

## Этап 8: Security-ревью

**Роль**: Проверить безопасность нового кода

- Инъекции (SQL, NoSQL, command injection)
- XSS в выводе
- Авторизация и права доступа
- Валидация входных данных

**Результат:** список уязвимостей или чисто

**"Безопасность проверена. Запускаем code review?"**

---

## Этап 9: Code Review (скилл `/review`)

**Роль**: Финальная проверка всего кода перед деплоем

**Запустить скилл `/review`** на всех созданных/изменённых файлах.

**Вердикт от `/review`:** можно / с оговорками / нельзя

**"Review завершён. Переходим к деплой-чеклисту?"**

---

## Этап 10: DevOps

**Роль**: Подготовить к запуску

- Чек-лист деплоя:
  - [ ] Бэкап БД
  - [ ] Бэкап файлов
  - [ ] Миграции БД (если есть)
  - [ ] Конфиги (если менялись)
  - [ ] Зависимости (npm install / composer install / go mod tidy и т.д.)
  - [ ] Порядок выкатки файлов
  - [ ] Откат: как вернуть назад если что-то пойдёт не так
  - [ ] Проверка после деплоя

**"Всё готово к деплою. Обновляем документацию?"**

---

## Этап 11: Документация (обновление)

**Роль**: Обновить документацию по факту реализации

- Сравнить документацию из Этапа 3 с реальным кодом
- Обновить если реализация отличается от плана
- Добавить `dev-guide.md`
- Добавить changelog запись

**"Документация обновлена. Собираем финальный отчёт?"**

---

## Этап 12: Финальный отчёт

Собрать итог по всем этапам:
```
Задача: [описание]
Тип: [новый модуль / доработка / багфикс]
Файлы создано: X
Файлы изменено: X
Тесты: X
Безопасность: ок / проблемы
Готовность к деплою: ок / с оговорками / нет
```

Сохранить в `_audit/build_[название]_YYYY-MM-DD.md`

---

## Правила оркестратора
1. **На каждом этапе спрашивай** — "Переходим к следующему?" / "Нужны правки?"
2. **Если пользователь говорит "пропусти"** — пропускай этап
3. **Если нашёл блокер** — останови и сообщи сразу
4. **Не редактируй файлы** без показа чек-листа и одобрения
5. **Запоминай контекст** — каждый следующий этап использует результаты предыдущего
6. **Оркестратор НЕ кодит сам** — он планирует, координирует, делегирует
