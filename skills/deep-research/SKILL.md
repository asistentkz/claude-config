## Фаза 3: RETRIEVE — Поиск информации

### Шаг 0: Определи стратегию поиска по типу темы

| Тип темы | Сигналы | Приоритетные источники | Паттерны запросов |
|----------|---------|----------------------|-------------------|
| Технология / инструмент | "что такое X", "как работает X" | docs, GitHub README, changelogs, release notes | `X documentation`, `X getting started`, `X vs alternatives` |
| Сравнение X vs Y | "сравни", "что лучше", "выбрать между" | бенчмарки, Reddit, HN, case studies, миграции | `X vs Y benchmark 2025`, `migrate from X to Y`, `X vs Y reddit` |
| Архитектура / паттерн | "как спроектировать", "архитектура для" | conference talks, блоги CTO, ThoughtWorks Radar, Martin Fowler | `X architecture pattern`, `X at scale`, `X production experience` |
| Безопасность | "уязвимости", "безопасность", "CVE" | NVD/CVE databases, OWASP, security advisories, GitHub Security | `X CVE`, `X security vulnerability`, `X OWASP` |
| Рынок / бизнес | "рынок", "конкуренты", "стоимость", "pricing" | аналитические отчёты, Crunchbase, Product Hunt, новости | `X market share`, `X pricing 2025`, `X competitors`, `X revenue` |
| Интеграция / API | "подключить", "интеграция", "API" | docs API, SDK, GitHub examples, Stack Overflow | `X API reference`, `X SDK example`, `X integration tutorial` |

Определи тип темы → используй соответствующие паттерны запросов для субагентов.
Если тема смешанная — комбинируй стратегии.

### Quick: последовательный поиск
3-5 запросов WebSearch последовательно по паттернам выбранной стратегии. Без субагентов.

### Standard+: параллельные субагенты

Запусти субагентов **одним сообщением** через Agent tool (все параллельно):

| # | Направление | Режимы | Описание |
|---|-------------|--------|----------|
| 1 | Основа | Standard+ | Официальная документация, определения, главные факты |
| 2 | Альтернативы | Standard+ | Конкуренты, сравнения, "X vs Y" |
| 3 | Проблемы | Standard+ | Баги, ограничения, негативный опыт, подводные камни |
| 4 | Практика | Deep+ | Case studies, примеры реализации, отзывы, миграции |
| 5 | Глубина | UltraDeep | Бенчмарки, академические источники, тренды, прогнозы |

### Промпт для каждого субагента

Используй этот шаблон, подставляя направление:

```
Ты — исследователь. Тема: {тема}. Направление: {направление}.
Временной фокус: {временной_фокус}.

Задача:
1. Сделай 3-5 запросов WebSearch по своему направлению
2. Для каждого найденного источника верни структурированный блок

Формат ответа — СТРОГО:

### Источник 1
- URL: [ссылка]
- Тип: [docs / блог / форум / GitHub / paper / news]
- Дата: [дата публикации или "неизвестно"]
- Ключевые факты:
  - [факт 1]
  - [факт 2]
  - ...

### Источник 2
...

В конце — раздел:
### Итого по направлению
- Найдено источников: [N]
- Главный вывод: [1-2 предложения]
- Пробелы: [что НЕ удалось найти]
```

### Сборка результатов

После завершения всех субагентов:
1. Собери ответы всех агентов
2. Пронумеруй все источники единой нумерацией [1], [2], [3]...
3. Удали дубликаты (один URL = один источник)
4. Отметь пробелы — если агент не нашёл данных по направлению, зафиксируй это для фазы CRITIQUE

## Фаза 4: EVALUATE — Оценка (Standard+)

### Оценка надёжности каждого источника

Присвой каждому источнику балл по шкале:

| Тип источника | Диапазон | Пример |
|---------------|----------|--------|
| Официальная документация | 90-100 | docs.docker.com, nodejs.org/docs |
| Peer-reviewed / академическая статья | 85-95 | arXiv, IEEE, ACM |
| GitHub (maintainer, official repo) | 75-90 | Issues от core team, README |
| Авторитетный блог (автор — эксперт) | 60-80 | Blog CTO компании, ThoughtWorks |
| Stack Overflow (принятый ответ, >10 голосов) | 50-70 | Проверенные решения |
| Обычный блог / Medium / Dev.to | 40-60 | Без верификации |
| Reddit / форум без доказательств | 20-40 | Мнения без ссылок |

Корректировки балла:
- **Дата >2 лет назад**: −10
- **Подтверждено 2+ источниками**: +10
- **Противоречит большинству**: −15 (но сохрани — может быть ценным)

### Для каждого источника зафиксируй

```
[N] Название — URL
    Тип: docs | Надёжность: 95 | Дата: 2025-11
    Факты: ...
```

### В отчёт добавь таблицу источников

```markdown
## Источники
| # | Название | Тип | Надёжность | Дата | URL |
|---|---------|------|-----------|------|-----|
| [1] | ... | docs | 95 | 2025-11 | ... |
| [2] | ... | блог | 55 | 2024-06 | ... |
```

Источники с надёжностью <40 — пометь ⚠️ и не используй как единственное основание для выводов.

## Фаза 6: CRITIQUE — Критика (Deep+)

Проверь свои выводы от лица двух ролей. Для каждой роли — ответь на ВСЕ вопросы чек-листа.

### Роль 1: Скептик-практик
- [ ] Есть ли реальные примеры использования в продакшне?
- [ ] Какие подводные камни не упомянуты?
- [ ] Масштабируется ли решение? При какой нагрузке ломается?
- [ ] Какова реальная стоимость владения (TCO), а не только цена лицензии?
- [ ] Есть ли vendor lock-in?

### Роль 2: Адвокат дьявола
- [ ] Какие аргументы ПРОТИВ основной рекомендации?
- [ ] Какой альтернативный подход мог бы быть лучше?
- [ ] Не базируется ли вывод на единственном источнике?
- [ ] Нет ли confirmation bias — искал ли я данные ПРОТИВ своей гипотезы?
- [ ] Что изменится через 1-2 года? Рекомендация устареет?

### Формат вывода критики

```
### Критика
**Скептик**: [находки — что слабо, что не проверено]
**Адвокат дьявола**: [контраргументы — что против]
**Критические пробелы**: [что нужно доискать]
**Вердикт**: пробелов нет / найдены пробелы → loop-back
```

### Протокол Loop-Back

Если найден критический пробел (вывод без достаточных источников, пропущена альтернатива):
1. Сформулируй **delta-query** — конкретный поисковый запрос ТОЛЬКО по пробелу
2. Выполни 1-3 WebSearch по delta-query
3. Интегрируй новые данные в выводы
4. Повтори критику ТОЛЬКО по закрытому пробелу

**Максимум 2 итерации loop-back.** Если после 2 итераций пробел остался — зафиксируй его в секции "Ограничения" отчёта.

## Фаза 7: REFINE — Уточнение (Deep+)

Финальная проверка перед упаковкой отчёта:

### Чек-лист

- [ ] Каждое утверждение процитировано [N]?
- [ ] Нет выдуманных или placeholder ссылок?
- [ ] Противоречия между источниками — объяснены или убраны?
- [ ] Рекомендации конкретные ("используйте X потому что Y"), не абстрактные?
- [ ] Слабые аргументы (один источник, <50 надёжность) — убраны или помечены как неподтверждённые?
- [ ] Все пробелы из CRITIQUE либо закрыты, либо в "Ограничениях"?
- [ ] Отчёт ≥80% проза, не буллеты?

Если чек-лист не пройден — исправь перед переходом к PACKAGE / RED TEAM.

## Фаза 8: RED TEAM — Стресс-тест (UltraDeep)

Три персоны проверяют финальный отчёт. Каждая пишет свой вердикт.

### Персона 1: Скептик
**Вопрос**: "Что может НЕ работать? Какие риски замолчаны?"
- Проверяет: технические риски, edge cases, failure modes
- Вердикт: `✅ рисков не нашёл` / `⚠️ найдены риски: [список]`

### Персона 2: Критик
**Вопрос**: "Где confirmation bias? Какие альтернативы не рассмотрены?"
- Проверяет: полноту рассмотрения, пропущенные альтернативы, однобокость
- Вердикт: `✅ анализ сбалансирован` / `⚠️ пропущено: [список]`

### Персона 3: Практик
**Вопрос**: "Как это конкретно применить в нашем стеке и контексте?"
- Проверяет: применимость к проектам пользователя, конкретность шагов
- Вердикт: `✅ готово к применению` / `⚠️ нужна адаптация: [что именно]`

### Секция в отчёте

```markdown
## Стресс-тест
| Персона | Вердикт | Детали |
|---------|---------|--------|
| Скептик | ✅/⚠️ | ... |
| Критик | ✅/⚠️ | ... |
| Практик | ✅/⚠️ | ... |

**Итого**: [общий вердикт — готов к использованию / с оговорками / требует доработки]
```

## Сохранение отчёта

**Quick**: отчёт выдаётся только в чат, файл НЕ создаётся.

**Standard+**: сохранить отчёт в файл + показать Executive Summary в чат.

Путь: `_research/[тема-slug]_[YYYY-MM-DD].md`
- `[тема-slug]` — тема транслитом или латиницей, пробелы → `_`, lowercase
- Пример: `_research/pm2_vs_docker_2026-04-07.md`

Порядок:
1. Создай папку `_research/` если нет
2. Сохрани полный отчёт (все секции) в файл через Write
3. В чат выведи:
   ```
   Отчёт сохранён: _research/[имя_файла].md
   
   ## Executive Summary
   [краткое резюме]
   
   Полный отчёт: [путь к файлу]
   ```

## Правила
- **НИКОГДА не редактируй существующие файлы проекта** — только исследуй и создавай отчёты
- Каждое фактическое утверждение — со ссылкой на источник [N]
- Запрещено: выдуманные ссылки, placeholder-ы, утверждения без источника
- Отчёт ≥80% проза, не буллеты
- Рекомендации конкретные ("используйте X потому что Y"), не абстрактные ("рассмотрите возможность")
- Если тема про код/архитектуру ЭТОГО проекта — перенаправь на `/research`
