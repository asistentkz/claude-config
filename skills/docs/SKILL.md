Задокументируй: $ARGUMENTS

## Правила
- **Не редактируй код проекта** — только создавай файлы документации
- Документацию сохраняй в `_docs/` в корне проекта
- Пиши на русском
- Документация должна быть полезна разработчику который видит код впервые

## Порядок работы

### 1. Анализ (read-only)
- Прочитай код модуля
- Определи публичные функции, эндпоинты, классы
- Найди запросы к БД → определи таблицы и связи
- Найди внешние зависимости (API, библиотеки)

### 2. Генерация документации

#### Описание модуля (`_docs/[модуль]/README.md`)
```markdown
# [Название модуля]

## Назначение
Что делает, для кого, зачем

## Файловая структура
путь/файл — описание

## Зависимости
- Внешние API: ...
- Библиотеки: ...
- Модули проекта: ...
```

#### Схема БД (`_docs/[модуль]/database.md`)
```markdown
## Таблицы

### table_name
| Поле | Тип | Описание |
|------|-----|----------|

### Связи
table_a.field → table_b.field
```

#### API / Эндпоинты (`_docs/[модуль]/api.md`)
```markdown
## Эндпоинты

### POST /path
- Описание: ...
- Параметры: ...
- Ответ: ...
- Авторизация: да/нет
```

#### Бизнес-логика (`_docs/[модуль]/logic.md`)
```markdown
## Основные процессы

### [Процесс]
1. Пользователь делает X
2. Система проверяет Y
3. Записывает в таблицу Z
4. Возвращает результат
```

#### Инструкция разработчику (`_docs/[модуль]/dev-guide.md`)
```markdown
## Как запустить
## Как тестировать
## Частые проблемы
## Что нельзя трогать и почему
```
