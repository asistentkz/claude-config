# Скилл spec-to-code

Полный пайплайн от готового ТЗ до рабочего кода с тестами.
Оркестрирует: разбор → тесты (RED) → имплементация (GREEN) → review.

**Не путать с `/test-driven-development`** (Superpowers) — тот про дисциплину RED-GREEN-REFACTOR.
Этот скилл — про **оркестрацию всего пайплайна**, используя `/test-driven-development` как базу цикла.

## Этап 0 — Вход

1. Получить ТЗ: путь к markdown-файлу или inline-описание от пользователя.
2. Если ТЗ нет — предложить `/tz` для создания.
3. Определить стек проекта из `CLAUDE.md`:
   - Go → `go test ./...`, `go vet`
   - Node.js → `npm test` или `yarn test`
   - Python → `pytest` или `python -m unittest`
   - PHP → `phpunit`
   - Flutter/Dart → `flutter test`
4. Определить тестовую команду из `Makefile` / `package.json` / `pubspec.yaml`.
   Если есть `make run-tests` или аналог — использовать его.

## Этап 1 — Разбор ТЗ в критерии приёмки

Прочитать ТЗ. Выдать пользователю **5–15 тестируемых критериев**:
- Каждый — одно утверждение ("Если X, то Y")
- Разделить по типу: unit / integration / edge-case
- Если ТЗ недостаточно чёткое — задать уточняющие вопросы (см. правило "Сначала уточняющие вопросы")

**Остановиться. Дождаться апрува критериев.**

## Этап 2 — Генерация тестов (RED)

1. Создать тесты для каждого критерия в правильном месте по конвенции стека:
   - Go: `*_test.go` рядом с кодом
   - Node.js: `tests/` или `__tests__/`
   - Python: `tests/test_*.py`
2. **Не писать production-код**. Только тесты.
3. Запустить тестовую команду — **все тесты ДОЛЖНЫ упасть** (RED).
4. Показать пользователю:
   - Сколько тестов создано
   - Какие упали (должны все)
   - Есть ли компиляционные ошибки (норма на этом этапе)

**Остановиться. Дождаться апрува перед имплементацией.**

## Этап 3 — Итеративная имплементация (GREEN) — ПОЛУАВТОМАТ

Цикл итерации:
1. Написать **минимум кода** для прохождения одного-двух тестов
2. Запустить тестовую команду
3. Показать результат:
   - ✅ Прошло: N тестов
   - ❌ Упало: M тестов (какие)
   - Diff изменений в коде
4. Ждать команду от пользователя:
   - **"дальше"** — следующая итерация (1 тест)
   - **"беги дальше"** — автомат до зелёного / до следующей остановки / max 10 итераций
   - **"стоп"** — остановиться, вернуть контроль

### Автоматический режим ("беги дальше")
- После каждой итерации одно короткое summary (1–2 строки)
- Не ждать подтверждения между итерациями
- Обязательная остановка при:
  - Все тесты зелёные
  - 10 итераций пройдено
  - Один и тот же тест упал 3 раза подряд (застряли)
  - Архитектурное решение требует апрува (см. правила)

## Этап 4 — Все тесты зелёные

1. Финальный прогон всей тестовой команды
2. Прогон lint (`go vet`, `golangci-lint`, `ruff`, `eslint` — что применимо)
3. Показать пользователю:
   - Тесты: X прошло / 0 упало
   - Покрытие: Y% (если тестовая инфра считает)
   - Финальный diff
4. **Предложить `/review`**
5. Если `/review` нашёл критичные замечания:
   - Для багов → новый тест (регрессия) → вернуться в Этап 3
   - Для стилистики → применить и перепрогнать тесты

## Этап 5 — Финал

Короткий отчёт (markdown):
```
## Реализовано
- Критерий 1: [статус]
- Критерий 2: [статус]
...

## Тесты
- Создано: N
- Все зелёные: ✅
- Покрытие: X%

## /review
- Замечания: ...
- Применено: ...
- Отложено: ...

## Следующие шаги
- ...
```

## Жёсткие правила

- **НЕ имплементируй без падающего RED-теста** — это нарушение TDD
- **НЕ переходи в автомат без явной команды "беги дальше"**
- **Лимит автомата — 10 итераций**, потом обязательная остановка с отчётом
- **"Готово" = все тесты green + /review пройден**
- Для архитектурных решений (новая зависимость, изменение структуры пакетов, breaking change API) — **остановиться и спросить**
- Не меняй несколько вещей сразу в одной итерации — один тест → одна правка
- Если заблокирован 3 итерации подряд на одном тесте — **стоп, показать пользователю**, не крутиться в цикле
- Опирайся на `/test-driven-development` (Superpowers) для дисциплины RED-GREEN-REFACTOR внутри цикла

## Пример использования

```
Пользователь: /spec-to-code D:/claude/aiplus_docs/auth_jwt_tz.md

Claude:
Этап 0: стек Go, команда тестов: make run-tests, тестовый фреймворк testing.
Этап 1: разбил ТЗ на 8 критериев приёмки:
  1. Токен с валидным payload возвращает 200
  2. Токен с истёкшим exp возвращает 401
  3. ...
Апрув?

Пользователь: да

Claude: Создал 8 тестов в api/auth_test.go. Прогон: 8 failed (RED).
Компиляция: OK.
Переходим к имплементации?

Пользователь: беги дальше

Claude [итерация 1]: реализовал ValidateToken, 3 теста зелёные.
Claude [итерация 2]: добавил проверку exp, 5 зелёных.
...
Claude [итерация 5]: все 8 зелёных. Запускаю /review.
```
