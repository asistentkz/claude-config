Подними повторяющиеся принципы из скиллов в правила: $ARGUMENTS

Сканирует скиллы, находит принципы, встречающиеся в нескольких скиллах, и предлагает вынести их в правила — дописать в существующий файл, исправить устаревшее, добавить секцию или создать новый файл. **Применяет только по номерам после апрува пользователя.**

Принцип работы: *детерминированный сбор + суждение LLM* — Glob/Read собирают факты полностью, затем субагент перечитывает весь контекст и выдаёт вердикты.

## Когда запускать

- Периодическое обслуживание правил (раз в месяц / после установки новых скиллов).
- После `/skill-stocktake`, если он вскрыл паттерны, которым место в правилах.
- Когда правила ощущаются неполными относительно используемых скиллов.

## Куда смотрят правила

Существующие правила (читать целиком — они небольшие):
- `d:\claude\aiplus_IT\aiplus_vault\shared_claude\rules\*.md` (comments.md, commit.md, pr-size.md, using-aiplus.md и др.)
- `C:\Users\umidi\.claude\CLAUDE.md` — глобальные правила в свободной форме (секции «Жёсткие правила», «Автоматические триггеры»).

Новые правила создавать в `shared_claude\rules\` (проектные, общие для AiPlus). Глобальные привычки уровня поведения — предлагать в CLAUDE.md.

## Фаза 1 — Инвентаризация (детерминированный сбор)

1. Glob всех `SKILL.md` в обеих папках скиллов (глобальной и `shared_claude\skills\`).
2. Read всех файлов правил + обоих CLAUDE.md.
3. Выведи пользователю:

```
Дистилляция правил — Фаза 1: Инвентаризация
────────────────────────────────────────
Скиллов: {N} files
Правил:  {M} files
Перехожу к перекрёстному анализу...
```

## Фаза 2 — Перекрёстное чтение, сопоставление, вердикт (суждение LLM)

Сгруппируй скиллы в **тематические кластеры** по описаниям. Анализируй каждый кластер субагентом (Agent, `general-purpose`), передавая полный текст всех правил. После всех батчей слей кандидатов: убери дубли, перепроверь требование «2+ скилла» по всем батчам вместе (принцип, найденный по 1 скиллу в двух батчах = валиден).

**Промпт субагенту:**

````
Ты аналитик, перечитывающий скиллы, чтобы извлечь принципы, заслуживающие повышения в правила.

## Вход
- Скиллы: {полный текст скиллов этого батча}
- Существующие правила: {полный текст всех файлов правил + CLAUDE.md}

## Критерии включения кандидата (ВСЕ должны выполняться)
1. **В 2+ скиллах**: принцип из одного скилла остаётся в этом скилле.
2. **Меняет поведение**: формулируется как «делай X» / «не делай Y», а не «X важно».
3. **Явный риск нарушения**: что ломается, если игнорировать (1 предложение).
4. **Ещё не в правилах**: проверь весь текст правил, включая концепции, выраженные другими словами.

## Сопоставление и вердикт
- **Append**: добавить в существующую секцию существующего файла правил
- **Revise**: содержание правила неточно/недостаточно — предложить исправление
- **New Section**: новая секция в существующем файле
- **New File**: создать новый файл правил
- **Already Covered**: уже покрыто (пусть и другими словами)
- **Too Specific**: должно остаться на уровне скилла

## Формат вывода (на кандидата) — JSON
{
  "principle": "1-2 предложения в форме «делай X» / «не делай Y»",
  "evidence": ["skill-name: §Секция", "skill-name: §Секция"],
  "violation_risk": "1 предложение",
  "verdict": "Append / Revise / New Section / New File / Already Covered / Too Specific",
  "target_rule": "имя файла §Секция, или 'new'",
  "confidence": "high / medium / low",
  "draft": "Черновик текста для Append/New Section/New File",
  "revision": { "reason": "...", "before": "...", "after": "..." }   // только для Revise
}

## Исключить
- Очевидные принципы, уже в правилах
- Знания по языку/фреймворку (это в скиллы или языковые правила)
- Примеры кода и команды (это в скиллы)
````

## Фаза 3 — Ревью и применение (только по апруву)

Сводная таблица:

```
# Дистилляция правил

Скиллов: {N} | Правил: {M} files | Кандидатов: {K}

| # | Принцип | Вердикт | Цель | Увер. |
|---|---------|---------|------|-------|
| 1 | ... | Append | comments.md | high |
| 2 | ... | New Section | new: agent-loops.md | medium |
```

Под таблицей — детали по каждому кандидату: evidence, violation_risk, черновик текста (для Append/New Section/New File) или before/after (для Revise).

Пользователь отвечает номерами:
- **Approve**: применить черновик как есть
- **Modify**: поправить черновик перед применением
- **Skip**: не применять

**Никогда не правь правила автоматически. Всегда требуй апрув пользователя.** Черновик должен содержать ссылку `См. скилл: [имя]`, чтобы читатель нашёл подробное «как».

## Принципы дизайна

- **Что, а не как**: в правила — только принципы. Примеры кода и команды остаются в скиллах.
- **Ссылка назад**: черновик включает `См. скилл: [имя]`.
- **Детерминированный сбор, суждение LLM**: Glob/Read гарантируют полноту, LLM — понимание контекста.
- **Защита от абстракций**: тройной фильтр (2+ скилла, тест на изменение поведения, риск нарушения) не пускает слишком абстрактные принципы в правила.
