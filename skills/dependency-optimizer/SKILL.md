Проведи аудит зависимостей: $ARGUMENTS

## Правила
- **НИКОГДА не удаляй и не обновляй пакеты без одобрения** — только анализ и рекомендации
- Проверяй фактическое использование через Grep, а не только наличие в package.json
- Учитывай что пакет может использоваться неявно (peer dependency, plugin)

## Шаг 1: Сбор информации

### Найди все манифесты зависимостей
- `package.json` — Node.js
- `go.mod` / `go.sum` — Go
- `composer.json` / `composer.lock` — PHP
- `requirements.txt` / `pyproject.toml` — Python
- `pubspec.yaml` — Flutter/Dart

### Для каждого манифеста собери
- Список dependencies и devDependencies
- Текущие версии
- Lock-файл (package-lock.json, yarn.lock)

## Шаг 2: Проверка уязвимостей

Запусти (если доступно):
- `npm audit --json` — Node.js
- `pip audit` — Python
- `composer audit` — PHP

Если команды недоступны — проверь через WebSearch:
- CVE для ключевых зависимостей
- Known vulnerabilities для устаревших версий

### Классификация
| Уровень | Описание |
|---------|----------|
| Critical | Remote code execution, auth bypass |
| High | XSS, SQL injection, data exposure |
| Moderate | DoS, information disclosure |
| Low | Minor issues |

## Шаг 3: Неиспользуемые зависимости

Для каждого пакета из dependencies:
1. Grep по проекту: `require('пакет')`, `import ... from 'пакет'`, `from пакет import`
2. Если не найдено прямого использования — проверь:
   - Используется ли как CLI (`scripts` в package.json)
   - Является ли peer dependency другого пакета
   - Загружается ли динамически
3. Если точно не используется — пометить для удаления

## Шаг 4: Устаревшие версии

Проверь ключевые зависимости:
- Есть ли мажорные обновления (breaking changes)
- Есть ли минорные/патч обновления (безопасно обновить)
- End of Life статус (Node.js версия, фреймворки)

## Шаг 5: Тяжёлые зависимости и замены

Известные замены:
| Тяжёлый пакет | Размер | Лёгкая замена | Размер |
|---------------|--------|---------------|--------|
| moment.js | 290KB | dayjs | 7KB |
| lodash | 530KB | lodash-es / нативные | 0-70KB |
| request | deprecated | node-fetch / undici | — |
| express-validator | — | zod | — |
| uuid | 18KB | crypto.randomUUID() | 0KB |

Проверь есть ли в проекте тяжёлые пакеты с лёгкими альтернативами.

## Шаг 6: Дубликаты между проектами

Если анализируем несколько проектов/ботов:
- Какие пакеты повторяются во всех/многих
- Можно ли вынести общие в shared модуль
- Разные версии одного пакета в разных ботах

## Формат отчёта

### Общая статистика
| Метрика | Значение |
|---------|----------|
| Проектов проанализировано | N |
| Всего зависимостей | N |
| С уязвимостями | N |
| Неиспользуемых | N |
| Устаревших (мажорно) | N |
| Тяжёлых с заменой | N |

### 🔴 Уязвимости
| # | Пакет | Версия | CVE / Описание | Severity | Проект | Решение |

### 🗑️ Неиспользуемые (можно удалить)
| # | Пакет | Проект | Уверенность |

### 🔄 Устаревшие
| # | Пакет | Текущая | Последняя | Breaking? | Проект |

### 📦 Тяжёлые → лёгкие замены
| # | Пакет | Размер | Замена | Размер | Проект |

### 🔁 Дубликаты между проектами
| # | Пакет | Проекты | Версии |

### Рекомендованные действия (по приоритету)
1. **Критично**: уязвимости → обновить/заменить
2. **Высокий**: неиспользуемые → удалить
3. **Средний**: тяжёлые → заменить на лёгкие
4. **Низкий**: устаревшие → обновить при возможности
