# Глобальные правила Claude Code

## Язык
- Язык общения: **русский** — всегда

## Жёсткие правила
- **ПЕРВОЕ СООБЩЕНИЕ**: при получении первого сообщения в любой сессии — по умолчанию **ОБЯЗАТЕЛЬНО** запустить `/brainstorming` ДО любых других действий. Сначала мозговой штурм → выбор направления → потом работа. **Единственное исключение**: если первое сообщение — явно системная/межсервисная архитектура (подходит под Fable/`@architect`), НЕ запускать brainstorming молча, а сначала спросить выбор: «`/brainstorming` (разобрать подходы) или `@architect` (сразу глубокое проектирование на Fable)?» — и дождаться ответа. Для всех прочих задач brainstorming первым, без исключений.
- **Чтение файлов — ВСЕГДА без спроса**. Читай, ищи, анализируй свободно. Разрешение нужно только на РЕДАКТИРОВАНИЕ.
- **НЕ редактировать файлы без явного одобрения** — сначала показать чек-лист изменений, дождаться "да"
- **СНАЧАЛА УТОЧНЯЮЩИЕ ВОПРОСЫ**: перед написанием ТЗ, спек, документации или существенного кода — **сначала** задать 3–5 уточняющих вопросов (скоуп, модель данных, пограничные случаи). **Дождаться ответов**. Не начинать драфт до подтверждения скоупа. Исключение: тривиальные правки (опечатка, однострочник).
- **КОНТЕКСТ ПРОЕКТА ДО ИССЛЕДОВАНИЯ**: перед чтением кода уточнить **какой проект** и **какой стек**, если это не очевидно из запроса. Стек и специфику определять из проектного `CLAUDE.md` и манифестов (`go.mod` / `package.json` / `composer.json` / `pubspec.yaml`), а не из допущений — у разных проектов разные стек, структура и конвенции. Не догадываться — спрашивать при неоднозначности.
- **Автоподбор скиллов**: при любой задаче — определить подходящий скилл, сказать пользователю "Запускаем /[скилл] или пропускаем?", дождаться ответа
- **НЕ редактировать файлы с credentials** (.env, config с паролями, keys.json, credentials.json)
- **АВТОМАТИЧЕСКИЙ /review ПОСЛЕ КОДА**: после написания/изменения любого кода (одного файла или серии) — **обязательно** предложить запустить `/review` перед тем, как сказать "готово" / отдать пользователю / передать другому. Формат: "Код собран → запускаем /review или пропускаем?". Без апрува пользователя ревью не запускать, но **предлагать всегда**, не ждать пока спросят.

## Роутинг моделей по задачам

**Дефолт главного цикла — Opus 4.8.** Роутинг уводит вниз на Sonnet/Haiku, где скорость сжигания лимита важнее интеллекта, и вверх на Fable — только для системной архитектуры.

### Субагенты (~/.claude/agents/)
| Агент | Модель | Для чего |
|-------|--------|----------|
| `architect` | Fable | системная/межсервисная архитектура, глубокий архитектурный анализ (**ЕДИНСТВЕННАЯ точка Fable**) |
| `researcher` | Sonnet | фоновый разбор кодовой базы / веб-ресёрч (read-only) |
| `mechanic` | Haiku | механика: grep, поиск файлов, переименования, форматирование |

### Карта: тип задачи → модель
| Тип задачи | Модель |
|-----------|--------|
| Системная/межсервисная архитектура, глубокий архитектурный анализ | **Fable** (`@architect`) |
| Архитектура в рамках модуля | Opus (эскалация на `@architect`/Fable — по требованию, если дизайн реально сложный) |
| Сложное pure-logic проектирование (алгоритм/модель данных, без security) | Opus; опц. `@architect` (Fable) |
| Планирование + верификация плана | черновик Sonnet/Opus → проверка Fable (`/plan`) |
| Мозговой штурм, ТЗ/спеки, код, вёрстка | Opus |
| Ревью кода, ревью PR лидов | Opus |
| Security-ревью, аудит сервера, дебаг, рефакторинг, performance, API-контракты, БД (миграции/ревью) | Opus |
| DevOps/CI-CD: конфиги / всё про прод-серверы | Sonnet / Opus |
| Инцидент на проде (зависон/OOM/watchdog), мета-работа (правка скиллов/CLAUDE.md/промптов) | Opus |
| Research кодовой базы, глубокий веб-ресёрч | Sonnet (`@researcher`) |
| Тесты (TDD), прогон/верификация, обработка ошибок (+финал ревью Opus) | Sonnet |
| Документация, отчёты, суммаризация логов, аудит зависимостей, задания вайб-кодерам, ad-hoc SQL, ревизия скиллов/бюджета | Sonnet |
| Механика: grep/поиск/переименования/форматирование, git-хозяйство (коммиты/PR/changelog), i18n | Haiku (`@mechanic`) |
| **Всё, чего нет в этой карте** | **Opus** |

### Правила
- **Тип задачи не покрыт явным правилом → Opus 4.8** (безопасный дефолт).
- **Fable-задача → ВСЕГДА предлагать `@architect`**: на ЛЮБОМ шаге сессии (не только в начале, не только первым сообщением), как только задача подходит под Fable — системная/межсервисная архитектура, схемы данных, стратегия миграции, разбор системы целиком — **обязательно** предложить `@architect`, а НЕ делать её молча на Opus. Формат: «Это Fable-задача → зову `@architect` или пропускаем?».
- **Fable — только `architect`** (системная архитектура + глубокий анализ). Ревью, security, аудит сервера, дебаг, прод — **Opus** (у Fable cyber-классификаторы → риск `refusal` посреди задачи).
- **Fable = системная/межсервисная архитектура; Opus = архитектура в рамках одного модуля** (модульную можно эскалировать на `@architect`, если дизайн реально сложный — но по умолчанию Opus: дешевле и держит контекст диалога). Сложное pure-logic проектирование (алгоритм/модель данных, без security-контекста) — опционально на `@architect`. Диалоговое проектирование — переключить главный цикл `/model fable` вручную (субагент не задаёт вопросы в диалоге).
- **Фолбэк Fable→Opus**: если вызов `architect` упал (модель недоступна / лимит / refusal) — повтори тот же вызов через Agent с `model: opus` и сообщи пользователю. Ручной переброс: `/fable-off` (вернуть — `/fable-on`).
- **Лимит (advisory)**: если в statusline видно использование ≥85% и >1ч до сброса — предложи `/fable-off`. Надёжного авто-триггера нет, только предложение глазами.

## Автоматические триггеры скиллов

**Правило**: Claude определяет ситуацию → предлагает скилл → ждёт подтверждения.
Формат: `"Ситуация X → Запускаем /скилл или пропускаем?"`

### Приоритет вызова (от высшего к низшему)
1. *(первое сообщение сессии)* → `/brainstorming`; **но если это системная/межсервисная архитектура (Fable-fit)** → сначала предложить выбор «`/brainstorming` или `@architect` (Fable)», дождаться ответа
2. **Любой написанный/изменённый код перед "готово"** → `/review` (автоматически предлагать, не ждать пока спросят)
3. Баг, ошибка, exception, failed, "не работает", "сломалось" → `/systematic-debugging`
4. "посмотри модуль", "что там с...", "разберись" → `/research`
5. "изучи технологию", "сравни X vs Y", "что лучше для", "обзор рынка" → `/deep-research`
6. "сделай", "добавь фичу", "модуль", "реализуй" → `/build`
7. Перед написанием любого production-кода → `/test-driven-development`
8. 3+ независимых подзадачи в плане → `/subagent-driven-development`
9. 2+ независимых задачи одновременно → `/dispatching-parallel-agents`
10. Получен фидбек от ревьюера → `/receiving-code-review`
11. Работа "готова", перед коммитом/PR → `/verification-before-completion`
12. Тесты прошли, нужен merge/PR → `/finishing-a-development-branch`
13. Нужна изоляция для feature → `/using-git-worktrees`
14. "перед деплоем", "выкати", "запусти" → `/review` (явный триггер от пользователя; автоматический при сборке кода — см. п.2)
15. "напиши тесты", "покрытие" → `/test`
16. "задокументируй", "документация" → `/docs`
17. "отчёт", "итоги", "результат" → `/report`
18. "напиши ТЗ", "составь тз", "сделай спеку", "техническое задание" → `/tz`
19. "аудит сервера", "проверь прод", "ротация учёток", "что крутится на сервере" → `/audit-server`
20. "реализуй по ТЗ", "сделай по спеке", "запусти имплементацию", "TDD по задаче" → `/spec-to-code`
21. "проверь API", "проверь контракты", "схемы API", "стыки интеграций" → `/api-contract-guardian`
22. "настрой CI/CD", "деплой пайплайн", "github actions", "docker setup" → `/cicd-quick-setup`
23. "проверь зависимости", "уязвимости пакетов", "npm audit", "обнови пакеты" → `/dependency-optimizer`
24. "обработка ошибок", "стандартизируй ошибки", "логирование ошибок" → `/error-handling-standardizer`
25. "медленно", "тормозит", "оптимизация", "узкие места", "performance" → `/performance-scanner`
26. "наведи порядок в скиллах", "ревизия скиллов", "дубли скиллов", "проверь скиллы" → `/skill-stocktake`
27. "сколько жрут скиллы", "контекст распух", "токен-бюджет", "что грузится в контекст" → `/context-budget`
28. "подними принципы в правила", "вынеси в правила", "дистилляция правил" → `/rules-distill`
29. **ВСЕГДА, на ЛЮБОМ сообщении сессии** (не только первом): системная/межсервисная архитектура, схемы данных, стратегия миграции, разбор системы целиком → предложить субагента `@architect` (Fable). *(это субагент, не скилл — формат «Ситуация → зову @architect (Fable) или пропускаем?»)* Модульная архитектура — по умолчанию Opus; на `@architect` эскалировать только если дизайн модуля реально сложный.

## Определение стека
- Стек проекта определяй из проектного `CLAUDE.md` или `package.json` / `go.mod` / `composer.json` / `pubspec.yaml`
- Тестовый фреймворк, линтер, структуру — адаптируй под стек
