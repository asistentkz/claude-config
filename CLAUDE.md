# Глобальные правила Claude Code

## Язык
- Язык общения: **русский** — всегда

## Жёсткие правила
- **ПЕРВОЕ СООБЩЕНИЕ**: при получении первого сообщения в любой сессии — **ОБЯЗАТЕЛЬНО** запустить `/brainstorming` ДО любых других действий. Без исключений. Сначала мозговой штурм → выбор направления → потом работа.
- **Чтение файлов — ВСЕГДА без спроса**. Читай, ищи, анализируй свободно. Разрешение нужно только на РЕДАКТИРОВАНИЕ.
- **НЕ редактировать файлы без явного одобрения** — сначала показать чек-лист изменений, дождаться "да"
- **Автоподбор скиллов**: при любой задаче — определить подходящий скилл, сказать пользователю "Запускаем /[скилл] или пропускаем?", дождаться ответа
- **НЕ редактировать файлы с credentials** (.env, config с паролями, keys.json, credentials.json)

## Глобальные скиллы
| Скилл | Когда использовать |
|-------|-------------------|
| `/brainstorming` | **ОБЯЗАТЕЛЬНО первое сообщение** — мозговой штурм, подходы, выбор направления |
| `/build` | **Оркестратор** — полный цикл: БА → Архитектура → Docs → TDD → Код → Review → DevOps |
| `/research` | Исследование модуля, архитектуры, зависимостей |
| `/review` | Ревью перед деплоем — блокеры, побочные эффекты |
| `/docs` | Документация для модуля — API, БД, бизнес-логика, dev-guide |
| `/report` | Итоговый отчёт по модулю |
| `/test` | Написание тестов для модуля/функции (TDD) |

## Автоматические триггеры скиллов
- *(первое сообщение сессии)* → `/brainstorming`
- "посмотри модуль", "что там с..." → `/research`
- "перед деплоем", "выкати", "запусти" → `/review`
- "тесты", "покрытие" → `/test`
- "отчёт", "итоги", "результат" → `/report`
- "сделай модуль", "добавь фичу", "исправь баг", "не работает" → `/build`
- "задокументируй", "документация" → `/docs`

## Определение стека
- Стек проекта определяй из проектного `CLAUDE.md` или `package.json` / `go.mod` / `composer.json` / `pubspec.yaml`
- Тестовый фреймворк, линтер, структуру — адаптируй под стек
