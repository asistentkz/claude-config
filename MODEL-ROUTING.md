# 🎛️ Роутинг моделей в Claude Code

Гайд по настройке автоматического распределения задач по моделям. Можно повторить у себя за 5 шагов (раздел в конце).

**Идея:** умная/дорогая модель дирижирует, дешёвые/быстрые — на подхвате. Экономим лимит там, где интеллект не критичен, и бьём топ-моделью только по сложному.

- **Opus 4.8** — дефолт главного цикла и всё, чего нет в карте.
- **Fable** — только системная архитектура (у Fable cyber-классификаторы → риск отказа на security/аудите, поэтому туда её НЕ пускаем).
- **Sonnet** — исследование, тесты, доки, отчёты, конфиги.
- **Haiku** — чистая механика и git-хозяйство.

---

## 1. Карта «тип задачи → модель»

| Тип задачи | Модель |
|-----------|--------|
| Системная/межсервисная архитектура, глубокий арх-анализ | **Fable** (субагент `architect`) |
| Архитектура в рамках одного модуля | Opus |
| Планирование + верификация плана | черновик Sonnet/Opus → проверка Fable (`/plan`) |
| Мозговой штурм, ТЗ/спеки, написание кода, вёрстка | Opus |
| Ревью кода, ревью PR | Opus |
| Security-ревью, аудит сервера, дебаг, рефакторинг, performance, API-контракты, БД (миграции/ревью) | Opus |
| DevOps/CI-CD: конфиги, всё про прод-серверы | Sonnet / Opus |
| Инцидент на проде (зависон/OOM), мета-работа (правка правил/промптов) | Opus |
| Research кодовой базы, глубокий веб-ресёрч | Sonnet (субагент `researcher`) |
| Тесты (TDD), прогон, обработка ошибок | Sonnet |
| Документация, отчёты, суммаризация, аудит зависимостей, ревизия скиллов | Sonnet |
| Механика: grep/поиск/переименования/форматирование, git-хозяйство, i18n | Haiku (субагент `mechanic`) |
| **Всё остальное** | **Opus** |

## 2. Три субагента (`~/.claude/agents/*.md`)

Модель зашита в frontmatter — вызываешь субагента, он сам поднимается на нужной модели.

| Агент | Модель | Для чего | Права |
|-------|--------|----------|-------|
| `architect` | Fable | системная архитектура, trade-offs, стратегии миграций | read-only |
| `researcher` | Sonnet | «что где лежит и как работает», сбор фактов, зависимости | read-only |
| `mechanic` | Haiku | grep, переименования, форматирование по чёткой инструкции | read + edit |

Пример frontmatter агента:

```markdown
---
name: architect
description: Системная и межсервисная архитектура, глубокий арх-анализ. Read-only.
model: fable
tools: Read, Grep, Glob, WebSearch, WebFetch
---
Ты — системный архитектор. ...системный промпт...
```

---

## 3. Пины моделей на скиллах (`model:` в frontmatter)

**Принцип:** пиним только там, где модель ОТЛИЧАЕТСЯ от Opus-дефолта.

- `model: sonnet` → `docs`, `report`, `test`, `research`, `deep-research`, `dependency-optimizer`, `cicd-quick-setup`, `error-handling-standardizer`, `context-budget`, `skill-stocktake`
- `model: opus` → `review` (флагман, явно)
- `model: haiku` → `fable-off`, `fable-on`
- **Без пина** (едут на дефолтном Opus): оркестраторы (`build`, `plan`, `spec-to-code`) и дисциплины (`systematic-debugging`, TDD, verification) — они сами дирижируют/судят.

---

## 4. Фолбэк, если Fable отвалилась

- `/fable-off` — переводит `architect` на Opus.
- `/fable-on` — возвращает на Fable.
- **Авто-ретрай**: если вызов `architect` упал (лимит/недоступна/отказ) — повтор на Opus.
- **Правило лимита** (глазами): использование ≥85% и >1ч до сброса → имеет смысл `/fable-off`.

---

## 5. Как повторить у себя — 5 шагов

1. **Дефолт главного цикла = Opus** — в `~/.claude/settings.json`: `"model": "opus[1m]"`.
2. **Создать 3 файла агентов** в `~/.claude/agents/` с frontmatter `name / description / model / tools` + системный промпт. Алиасы `fable` / `opus` / `sonnet` / `haiku` в поле `model` работают (проверено).
3. **Проставить `model:`** в frontmatter нужных скиллов (см. п.3).
4. **Добавить в глобальный `CLAUDE.md`** секцию с картой из п.1 + правилами — чтобы главный цикл сам роутил задачи, которых нет в скиллах.
5. *(опц.)* **statusline** — показывает текущую модель в статус-строке, удобно контролировать, что роутинг сработал.

> 💡 Вызов конкретной модели можно форсить и разово: у Agent-tool есть параметр `model`, он перекрывает модель агента.
